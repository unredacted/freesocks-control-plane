# Secrets & environment runbook

Every secret/credential the self-hosted stack needs, **who generates it**, how to
rotate it, and its blast radius. The goal is a **seamless first deploy**: the only
values you ever type by hand are the ones that genuinely come from an external
system. Everything else is generated for you, idempotently.

Three homes for configuration:

- **`.env.beta`** — container-infra env, read by `docker-compose.beta.yml` (Caddy,
  the Convex backend identity, Postgres) **plus the CDN-blinding PUBLIC pins**
  (`VITE_FS_*`) baked into the SPA build.
- **`.env.convex`** — the Convex **deployment** env. The `deployer` service applies
  each line with `bunx convex env set` on `up`.
- **Convex deployment env directly** — what the backend reads at runtime
  (`process.env.*`). The `deployer` populates it from `.env.convex` + its own
  auto-generation.

> The five core app secrets are auto-generated by the **deployer**; the infra
> randoms + CDN-blinding keypair are generated by **`bun run bootstrap`**
> (`scripts/bootstrap-secrets.mjs`). Both are **generate-once / never-overwrite**,
> so re-running either is safe.

---

## 1. Auto-generated by the deployer (do nothing — leave `CHANGE_ME`)

`docker/deploy-entrypoint.sh` generates each of these ONCE if the deployment
doesn't already have it (checked via `bunx convex env list`), then never
regenerates. Leave them as `CHANGE_ME` in `.env.convex`.

| Var                         | Purpose                                                                            | Rotation / blast radius                                                                                                                  |
| --------------------------- | ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `SESSION_SIGNING_KEY`       | HMAC key for the member `fs_session` signed cookie                                 | Rotating invalidates **all member sessions** (everyone re-logs-in with their account number). Safe but disruptive.                       |
| `ADMIN_SESSION_SIGNING_KEY` | HMAC key for the admin `fs_admin_session` cookie                                   | Rotating signs out **all admins** (re-login with a passkey).                                                                             |
| `ADMIN_BOOTSTRAP_SECRET`    | Gates the first-admin passkey registration at `/admin`                             | Retrieve with `bunx convex env get ADMIN_BOOTSTRAP_SECRET`. Inert once any admin credential exists (bootstrap locks).                    |
| `IP_HASH_SALT`              | HMAC salt for per-IP rate-limit buckets + free-tier keying (never stores a raw IP) | Rotating resets rate-limit buckets (harmless) — but do it **once**; it is not a credential users hold.                                   |
| `ACCOUNT_ID_PEPPER`         | Peppered keyed hash of the 32-digit account number                                 | **SET ONCE.** Rotating **invalidates every account number** — every member is locked out permanently. Never rotate on a live deployment. |

## 2. Generated by `bun run bootstrap` (idempotent)

`scripts/bootstrap-secrets.mjs`, run once before `docker compose ... up`. Fills
only empty/`CHANGE_ME` placeholders; re-running never overwrites a set value.

**Infra randoms → `.env.beta`** (the compose `${VAR:?}` guards need them present
_before_ `up`):

| Var                 | Purpose                                                 | Notes                                                                        |
| ------------------- | ------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `INSTANCE_SECRET`   | Convex backend instance identity                        | Set once; changing it invalidates the admin key.                             |
| `POSTGRES_PASSWORD` | Postgres superuser password (interpolated into the DSN) | URL-safe hex. Changing it needs a matching DB change.                        |
| `CAP_ADMIN_KEY`     | Login to the self-hosted Cap captcha dashboard          | The bootstrap **prints it** (you need it to create the Cap site key/secret). |

**CDN-blinding keypair → split across both files, generated as a UNIT** so the
public pins always match the secret keys:

| Var                                    | Home                                     | Purpose                                                                 |
| -------------------------------------- | ---------------------------------------- | ----------------------------------------------------------------------- |
| `FS_SERVER_HPKE_SK`                    | `.env.convex` (secret)                   | X-Wing (X25519+ML-KEM-768) server seed — opens sealed login requests.   |
| `FS_MANIFEST_SK` / `FS_MANIFEST_SK_PQ` | `.env.convex` (secret)                   | Ed25519 + ML-DSA-65 manifest signing keys (the epoch-key trust anchor). |
| `VITE_FS_SERVER_HPKE_PK` / `_KID`      | `.env.beta` (public, baked into the SPA) | The HPKE public pin the client seals to.                                |
| `VITE_FS_MANIFEST_PK` / `_PK_PQ`       | `.env.beta` (public)                     | The manifest public keys the client verifies epoch keys against.        |

Running the bootstrap **enables CDN-blinding** (body-sealing the account number +
proxy URL from a compromised CDN). Not running it ships the SPA "dark" (plaintext
dual-mode over TLS) — safe, just unsealed. To **rotate** the keypair: clear
`FS_SERVER_HPKE_SK` in `.env.convex` AND the `VITE_FS_*` pins in `.env.beta`, re-run
the bootstrap, then rebuild `web` + redeploy. (Never edit one side alone — the
bootstrap warns and refuses if the two drift.)

## 3. External credentials — fill by hand (cannot be generated)

A `CHANGE_ME` on any of these **aborts the deployer** (so a forgotten one fails
fast). All live in `.env.convex` unless noted.

| Var(s)                                                                  | Where it comes from                                                                                                                           |
| ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `SITE_ADDRESS`, `ACME_EMAIL` (`.env.beta`)                              | Your domain + Let's Encrypt contact.                                                                                                          |
| `CAP_API_ENDPOINT`, `CAP_SITE_KEY`, `CAP_SECRET`, `CAP_PUBLIC_ENDPOINT` | The Cap dashboard (log in with `CAP_ADMIN_KEY`, create a site key on first boot).                                                             |
| `WEBAUTHN_RP_ID`, `WEBAUTHN_ORIGIN`                                     | Your deployment domain / page origin.                                                                                                         |
| `WEBHOOK_SIGNING_SECRET`                                                | Shared HMAC with a billing portal (only if one posts to `/api/webhooks/billing`). Until set, that route returns `503 webhook.not_configured`. |
| `NOWPAYMENTS_*`, `STRIPE_*`, `PAYPAL_*`, `PUBLIC_BASE_URL`              | Per billing rail (see `docs/billing.md`). Required once a rail is enabled in Admin → Billing.                                                 |
| `REMNAWAVE_BASE_URL`, `REMNAWAVE_API_TOKEN`                             | Optional one-time cutover seed of the first backend instance; afterwards managed in the admin CMS (`backendServers`).                         |

**Topology flags (keep as shipped):** `ENVIRONMENT=production` (Secure cookies),
`TRUSTED_PROXY=true` (behind Caddy), **`CF_FRONTED` unset** (no real Cloudflare
edge — setting it trusts a spoofable `cf-connecting-ip`). `POP_REQUIRED` /
`POP_EXPECTED_HOST` are CDN-blinding rollout toggles (see
`docs/threat-model-cdn-blinding.md`).

## First-deploy sequence

```sh
cp .env.beta.example   .env.beta
cp .env.convex.example .env.convex
bun run bootstrap                         # §1-adjacent infra randoms + §2 CDN-blinding keys
$EDITOR .env.beta .env.convex             # fill the §3 external creds
docker compose -f docker-compose.beta.yml --env-file .env.beta up -d --build
docker compose -f docker-compose.beta.yml --env-file .env.beta logs deployer  # "[deploy] OK"
```

The deployer then auto-generates the §1 core secrets, applies `.env.convex`, and
seeds tiers + settings. See `docs/beta-deploy.md` for the full runbook and
`docs/convex-self-hosting.md §5` for the exhaustive per-var reference.

## Never commit real secrets

`.env.beta` / `.env.convex` / `.env.local` are gitignored. Only the `*.example`
templates are committed. The CDN-blinding `VITE_FS_*` PUBLIC pins are safe to bake
into the (public) SPA bundle by design; the `FS_*` secret halves never leave the
Convex deployment env.
